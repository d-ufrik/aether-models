# Manifest format — `manifests/<model-id>.json`

A small JSON file per model describing where the weights live, how
big they are, and what the runtime defaults are. The catalog
(`catalog.xml`) is the **only** thing Aether.app reads from this
repo today — manifests are an intermediate artifact emitted by
`upload-to-r2.sh` and useful for tooling that wants a stable
machine-readable single-model record.

## Example — `qwopus-3.5-4b-coder-mtp-q4_k_m.json`

```json
{
  "id": "qwopus-3.5-4b-coder-mtp-q4_k_m",
  "engine": "llamacpp",
  "display_name": "Qwopus 3.5 4B Coder (MTP)",
  "license": "Apache-2.0",
  "download": {
    "url": "https://aether-models.ufrik.com/llamacpp/qwopus-3.5-4b-coder-mtp-q4_k_m/model.gguf",
    "size_bytes": 2783446560,
    "sha256": "50b426c393fc07aba6438d8f7d66bad156e06a6afb6bb80f15def9c313efa133"
  },
  "ram_min_gib": 16,
  "context_default": 32768,
  "mtp": {
    "available": true,
    "draft_layers_in_main_gguf": true
  }
}
```

## Field reference

| Field | Type | Required | Notes |
|---|---|---|---|
| `id` | string | yes | Globally unique within Aether. Slug form, lowercase, hyphens, no slashes. Used as the directory name under `~/Library/Application Support/Aether/models/<engine>/<id>/`. |
| `engine` | string | yes | One of `llamacpp`, `mlx`, `mtplx`. The companion picks the right runtime based on this. |
| `display_name` | string | yes | Shown in the Aether wizard / dashboard. |
| `license` | string | yes | SPDX identifier (`Apache-2.0`, `MIT`, `Gemma`, etc.) or upstream's license name. |
| `download.url` | string (URL) | yes | Public R2 URL. Aether downloads directly. |
| `download.size_bytes` | integer | yes | Exact size on disk after download. Used for the progress bar's denominator. |
| `download.sha256` | string (hex) | yes | Lowercase hex SHA-256 of the GGUF. Aether verifies before installing. |
| `ram_min_gib` | integer or null | no | Minimum recommended unified-memory for this model. The wizard's "Recommended" selector uses this to match a model to the detected RAM. |
| `context_default` | integer | yes | Initial `--ctx-size` for the llama-server / mlx-lm spawn. |
| `mtp.available` | boolean | no | If true, the model supports Multi-Token Prediction speculative decoding via llama.cpp's `--spec-type draft-mtp`. |
| `mtp.draft_layers_in_main_gguf` | boolean | no | If true, the draft layers ship inside the main GGUF (Jackrong / Qwopus pattern). If false, there's a separate drafter file URL. |
| `mtp.drafter_url` | string | conditional | Required when `draft_layers_in_main_gguf == false`. Public URL of the drafter GGUF. |
| `mtp.drafter_size_bytes` | integer | conditional | Same as above. |
| `mtp.drafter_sha256` | string | conditional | Same as above. |

## Lifecycle

Manifests are generated by `upload-to-r2.sh` at upload time (every
field above is filled from CLI flags + computed from the local
file). They are then **manually merged** into the higher-level
`catalog.xml` because the catalog has additional fields (taglines,
use-case categorization, vendor blurbs) that come from human
curation, not the upload script.

The catalog is the source of truth for the running app. Manifests
are convenience artifacts; deleting them locally doesn't break
anything for end users. They exist so external tooling can grep
for `"sha256":` without having to write an XML parser.
